# 2026-04-01

## Session 1: Encounter Sets System

Replaced the flat `reveal_odds` single-pool system with a two-stage roll using additive encounter sets, profile attributes, and an event system.

### New tables (migration 008)
- `encounter_sets` ‚Äî named creature pools with optional JSONB conditions
- `encounter_set_members` ‚Äî badge_id + weight per set, tier from `badges.rarity`
- `profile_attributes` ‚Äî global attribute definitions (key, label, data_type)
- `profile_attribute_values` ‚Äî per-user typed values (number/boolean/string columns)
- Migrated `badges.rarity` from 4-tier (common/uncommon/rare/legendary) to 5-tier (common/rare/epic/unique/legendary)
- Migrated existing `reveal_odds` entries into default encounter set, dropped `reveal_odds` table

### New domains
- **encounter-sets** ‚Äî CRUD + condition evaluator + qualifying sets + eligible creatures
- **profile-attributes** ‚Äî attribute definitions + per-user value upserts + `getUserAttributeMap`
- **events** ‚Äî lightweight in-process handler registry (`emit`/`registerHandler`)

### Two-stage roll mechanic
- Stage 1: tier roll against fixed constants (common 35%, rare 20%, epic 12%, unique 5%, legendary 3%, nothing 25%)
- Stage 2: weighted random among eligible creatures in winning tier from all qualifying sets
- Dedup: same creature in multiple qualifying sets ‚Üí use highest weight (not summed)
- Conditions evaluated against both event data (current log) and profile attributes (precomputed)

### Event system
- `emit('log:saved', ...)` fires after daily log save
- Handlers: `updateStreakAttribute`, `updateTotalLogsAttribute`, `updateBestProteinAttribute`, `updateHasLoggedWeightAttribute`
- Each handler upserts `profile_attribute_values` via admin client

### Admin UI
- `/admin/encounter-sets` ‚Äî list/create/delete sets, condition editor, member management
- `/admin/profile-attributes` ‚Äî list/create attribute definitions
- Deleted old `/admin/reveal` page and API route

### Wiring
- `POST /api/logs` now emits `log:saved` event after save (fire-and-forget)
- `POST /api/reveal/roll` now accepts optional `{ log }` body for event-source conditions
- `WoodlandReveal` component passes log data via `CustomEvent.detail` ‚Üí roll API body
- Updated admin hub links

### Key decisions
- Fixed tier constants in code, not admin-configurable ‚Äî rarity curve rarely changes
- Three typed columns (value_number/boolean/string) over JSONB for profile attribute values ‚Äî indexable, type-safe
- Event handlers run fire-and-forget from the route ‚Äî don't block the save response
- Profile attributes computed on save (not at roll time) ‚Äî fast rolls, single source of writes
- Tier always from `badges.rarity` ‚Äî no per-member tier override to avoid inconsistency
